Honor the [MODE] hint: in AUTO-ANALYZE, extract actual work items from the document; in DIRECT-INPUT, design useful work items for the topic and use any <<<CONTEXT>>> only as background (domain/terminology/tone).

# Output format (STRICT)
Return ONLY a JSON object — no prose, no markdown fences — matching this schema:

```
{
  "title": "<short board title, 1-80 chars>",
  "cards": [
    {
      "name": "<short card label, 1-60 chars>",
      "section": "<group/category this card belongs to, e.g. 기획 / 디자인 / 개발 / 출시>",
      "status": "todo" | "doing" | "review" | "blocked" | "done",
      "priority": "low" | "medium" | "high" | "urgent",
      "start": "YYYY-MM-DD",
      "due": "YYYY-MM-DD",
      "progress": <integer 0-100>
    },
    ...
  ]

```

# Kanban semantics
- A card is one concrete work item, not a vague theme. Prefer visible deliverables or actions.
- `todo` means not started / ready / backlog.
- `doing` means active work in progress. Keep WIP small.
- `review` means waiting for review, QA, approval, or feedback.
- `blocked` means work cannot proceed because of an explicit blocker: missing information, external dependency, approval wait, access issue, vendor wait, etc. Do NOT use `blocked` merely because a task is difficult.
- `done` means completed/shipped/accepted.

# Hard rules
1. Every card MUST have `name`, `section`, and `status`.
2. Use ONLY these statuses: `todo`, `doing`, `review`, `blocked`, `done`.
3. In DIRECT-INPUT mode for a new plan, most cards should be `todo`. Use `doing`, `review`, `blocked`, or `done` only when the topic/context clearly implies current state.
4. Respect WIP: normally use at most 3 `doing` cards and at most 3 `review` cards. If unsure, choose `todo`.
5. Use 2-5 sections for most boards. Sections are categories/phases, not statuses.
6. Keep it focused: 6-14 cards for most boards. Hard cap at 25.
7. Set `priority` only from real urgency, risk, dependencies, deadlines, or user emphasis. If unsure, use `medium`.
8. Use `urgent` sparingly for P0-style work: deadlines, launch blockers, incidents, or severe risk.
9. `progress` is completion 0-100. Omit it unless there is a clear basis. For `done`, use 100 or omit. For `todo`, omit or 0.
10. If schedule fields are requested in [GENERATION HINTS], add realistic `start`/`due` dates on or after [PROJECT START] where useful. Otherwise omit dates unless the source already contains them.
11. If using dates, `due` is the inclusive target/finish date and must not be before `start`.
12. Respond in [TARGET LANGUAGE] for `title`, `name`, and `section`. Status/priority/date values stay exactly as the schema requires.

# Style
- Names are short, concrete labels (2-7 words). Prefer noun phrases ("요구사항 정리", "API 구현") or short verb phrases.
- Sections are compact labels ("기획", "디자인", "개발", "QA", "출시") — not full sentences.
- Avoid duplicate cards and overlapping work. Make cards small enough to move across columns.
